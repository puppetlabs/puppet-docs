このセクションでは、*nixオペレーティングシステムについて取り上げます。 WindowsへのPuppet Enterpriseのインストールについては、「Windows agentのインストール」を参照してください。

Puppet Enterpriseをダウンロードする

まず、Puppet LabsのWebサイトから、最新バージョンのPuppet EnterpriseのtarballとGPG署名(「.asc」ファイル)をダウンロードします。

インストーラTarballを選択する

ご利用のOSバージョンとアーキテクチャに合ったPuppet Enterpriseのtarballを選択してください。

*nix向けtarball

ファイル名の末尾 インストール対象の環境
-el-<version and arch>.tar.gz RHEL、CentOS、Scientific Linux、またはOracle Linux
-ubuntu-<version and arch>.tar.gz Ubuntu LTS
-sles-<version and arch>.tar.gz SLES

注意: RHEL 5、6および7ではSELinuxのバインドを利用できます。 これらはデフォルトではインストールされませんが、puppet-agentパッケージには含まれています。

インストーラを検証する

Puppet Enterpriseインストーラを検証するには、Puppet Labsの公開鍵をインポートし、ダウンロードしたtarballの暗号検証を実行します。 Puppet Labsの公開鍵は、Puppet Labsによって証明されており、pgp.mit.eduなどの公開鍵サーバー、およびPuppet Labsから入手できます。 GnuPGをインストールし、PE tarballと共にGPG署名(「.asc」ファイル)をダウンロードする必要があります。

Puppet Labsの公開鍵をインポートするには、次のコマンドを実行します。 

wget -O - https://downloads.puppetlabs.com/puppetlabs-gpg-signing-key.pub | gpg --import

以下のような結果が表示されます。 

gpg: key 4BD6EC30: public key "Puppet Labs Release Key (Puppet Labs Release Key) <info@puppetlabs.com>" imported
gpg: Total number processed: 1
gpg:               imported: 1  (RSA: 1)

公開鍵のフィンガープリントを出力するには、次のコマンドを実行します。 

gpg --fingerprint 0x1054b7a24bd6ec30

検証の際に表示されるフィンガープリントと、ここで表示されるPuppetのフィンガープリントは完全に一致しなければなりません。 

Primary key fingerprint: 47B3 20EB 4C7C 375A A9DA  E1A0 1054 B7A2 4BD6 EC30

次に、以下のコマンドを実行してtarballのリリース署名を検証します。 

$ gpg --verify puppet-enterprise-<version>-<platform>.tar.gz.asc

以下のような結果が表示されます。 

gpg: Signature made Tue 18 Jun 2013 10:05:25 AM PDT using RSA key ID 4BD6EC30
gpg: Good signature from "Puppet Labs Release Key (Puppet Labs Release Key)"

注意: 署名を検証する際、リリース鍵の署名の1つへの信頼できるパスがない場合は、以下のような警告メッセージが表示されます。 

Could not find a valid trust path to the key.
    gpg: WARNING: This key is not certified with a trusted signature!
    gpg: There is no indication that the signature belongs to the owner.

これは、リリース鍵に署名したユーザーを証明できる信頼できるパスが作成されていないためです。この警告は無視してかまいません。

Puppet Enterpriseのインストール

以下では、Puppet Enterpriseをスムーズにインストールするために知っておくべきことを説明します。 Puppet Enterpriseの機能は複数のコンポーネントにより構成されています。インストーラを実行すると、これらすべてのコンポーネントがインストールされ、設定されます。 複数のコンポーネントを1つのノードにインストール（モノシックインストール）、もしくは複数のノードにコンポーネントを分散してインストール（分割インストール）することができます。ただしagentコンポーネントについては、すべてのノードにインストールされます。

インストールを始める前に、デプロイするものを決めておきましょう。 Puppet Enterpriseコンポーネントをインストールするすべてのノードにアクセスできるよう、各ノードの完全修飾ドメイン名を把握し、ファイアウォールのルールが必要なポートにアクセスできるよう設定されているか確認する必要があります。

これらの情報を確認した後、次の三つの手順でインストールを実行します。

  1. インストール方法を選択します。

  2. Puppet Enterpriseの主要コンポーネント、すなわち、Puppet master、PuppetDB (およびデータベースサポート)、PEコンソールをインストールします。

  3. Puppet Enterpriseによる管理を行うすべてのノードに、Puppet agentをインストールします。 「agentのインストール手順」を参照してください。

インストール方法の選択

開始する前に、インストール方法を選択します。 インストール方法は、以下の手順で決定することができます。

  • Webベースのインターフェイスで、画面に表示される手順に従ってインストールします。 これについてはインストール時の質問に答えることで、正しいインストール方法を選択できると思います。 SSH証明書を使えるのならば、この方法で素早くインストールして運用を開始できます。 以下のどちらのインストールを行うか選択します。

    • モノリシックインストール：標準モノリシックインストールまたはモノリシックインストールにコンパイルmasterを追加する場合の土台となるインストール方法。

    • 分割インストール：大規模インストールの土台となるインストール方法。

  • Webベースのインターフェイスで応答ファイルを作成した後、インストーラのスクリプトに引数を追加して、インストールを実行します(例: sudo ./puppet-enterprise-installer -a ~/my_answers.txt)。 応答ファイルを使用してPuppet Enterpriseをインストールする方法の概要については、「応答ファイルを用いたインストール」を参照してください。

  • PEのインストール用tarballで提供される応答ファイルを編集します。 手順については、「応答ファイルを用いたインストール 」を参照してください。

「システム要件」で、ハードウェア関連の仕様を確認します。

注意: インストールを開始する前に、「Enterpriseのコンポーネント」を読んで、Puppet Enterpriseのインストールに関連する部分をよく理解しておくとよいでしょう。

Puppet Enterpriseのコンポーネントについて

インストールを開始する前に、以下のPuppet Enterpriseのコンポーネントをよく理解しておく必要があります。

Puppet agent

Puppet agentは、パッケージマネージャを使ってインストールするのがもっとも簡単です(「agentのインストール」を参照)。 リモートリポジトリから直接パッケージをインストールできないプラットフォーム(Windows)の場合は、インストーラスクリプトを使用してください。

このコンポーネントは、デプロイ環境のすべてのノードにインストールする必要があります。 Puppet master、PuppetDB、コンソールコンポーネントがインストールされたサーバーには、Puppet agentコンポーネントも自動的にインストールされます。

Puppet agentコンポーネントをインストールしたノードでは、以下を実行することができます。

  • Puppet agentデーモンを実行し、Puppet masterから設定を受信して適用する。
  • MCollective関連のメッセージを聴取し、関連アクションを呼び出す。
  • PuppetDBが使用するデータをmasterに送信する。

Puppet master

ほとんどのデプロイ環境では、このコンポーネントは1つのノードにインストールすることを推奨します(複数のPuppet masterをインストールするには追加設定が必要ですが、このガイドでは取り上げません)。 Puppet masterには、堅牢な専用サーバーを使用する必要があります。 詳しくは「システム要件」を参照してください。

Puppet masterでは、以下を実行することができます。

  • Puppet agentノードの設定カタログをコンパイルして提供する。
  • 自身のActiveMQサーバー経由でオーケストレーションメッセージを経路指定する。
  • peadmin ユーザーとしてログインした管理者が) 有効なMCollective関連のコマンドを発行する。

注意: デフォルトではpe-puppetserverが再起動する度に、利用可能なアップデートの有無を確認します。 正確なアップデート情報を取得するため、masterはいくつかの基本的な情報を匿名でPuppet Labsのサーバーに対して送信します。 この動作は無効にできます。 収集されるデータ、アップグレードチェックを無効にする方法について、詳しくは「Puppet Server configuration」ページを参照してください。 利用可能なアップデートがある場合は、メッセージが表示されます。

PuppetDBおよびデータベースサポート

PuppetDBコンポーネントは、Puppet EnterpriseによってインストールされたPostgreSQL、またはユーザーが手動で設定したPostgreSQLのインスタンスを使用します。 モノリシックインストールの場合、PuppetDBは、コンソールおよびPuppet masterコンポーネントと同じノードにインストールされます。 分割インストールの場合、PuppetDBはPuppetDB用として利用するサーバーにインストールされます。 インストール中に、Puppet EnterpriseでPostgreSQLインスタンスをインストールするか、または設定済みのPostgreSQLインスタンスを使用するか確認するメッセージが表示されます。

ロールベースのアクセス制御(RBAC)のデータベースとノードに付随する情報を管理するノードクラシファー (node classifier )のデータベースも、PuppetDBと同じPostgreSQLのインスタンス上で動作します。

PuppetDBは、Puppet Enterpriseのデータをすばやく格納できる拡張性と信頼性の高いデータウェアハウスです。 Puppet Enterpriseによって生成されたデータをキャッシュに入れ、強力なAPIによって高度な機能を高速で提供します。

PuppetDBには以下が格納されます。

  • すべてのノードからの最新データ
  • すべてのノードの最新カタログ
  • すべてのノードに関する(構成可能なオプション)の14日分(設定可能)のイベントレポート

PuppetDBデータベースを手動で設定する場合は、「PuppetDBの設定マニュアル」で詳しい手順を確認してください。

重要: 既存のPostgreSQLインスタンスを使用する場合は、データベースサポートに使用するノードのホスト名とポート、データベースのアクセスに使用するユーザーパスワードが必要です。

Puppet Enterprise Webコンソール

分割インストールの場合、コンソールは専用サーバーにインストールしますが、モノリシックインストールでは、ほかのすべてのPEコンポーネントと同じサーバーにインストールします。

コンソールサーバーでは以下を実行できます。

  • コンソールWebインターフェイスを提供する。管理者はこのインターフェイスで、ノード上のリソースを直接編集したり、ノードをグループ化したり、クラスを割り当てたり、レポートやグラフ、インベントリ情報を確認したりできます。
  • レポートを収集し、ノード情報をPuppet masterに提供する。

ロールベースのアクセス制御(RBAC)

RBACによってPEノードはセグメント化されますので、適切なユーザーに対してタスクを安全に委任することができます。 たとえば、アプリケーションチーム全体でインフラをセグメント化し、ほかのアプリケーションに影響を与えることなく、自身のサーバーを管理できます。 さらに、RBACはMicrosoft Active DirectoryやOpenLDAPなどの標準ディレクトリサービスに直接接続できるため、ユーザーや認証を簡単に管理できます。

RBACの詳しい使い方については、「PEユーザーガイド」を参照してください。

Node Classifier (NC)

Node Classifier(ノードにクラスをアサインするサービス)を活用すれば、これまでにないパワフルで柔軟な方法でノードを整理して設定できます。 API駆動型のバックエンドサービスにより、直感的に操作できる新しいWeb UIを通じて、インフラを統合管理できる最新のアプローチが実現します。 グループレベルでクラスを割り当てたり、ユーザーが定義したルールに基づいてノードを動的にグループへマッチングさせることができます。

新しいノード分類子については、「PEユーザーガイド」を参照してください。

注意: PuppetDBと同様、 RBACとNode ClassifierのデータベースはPostgreSQL上で実行されます。既存のPostgreSQLインスタンスを使用することも、Puppet Enterpriseのインストール時に作成することもできます。 既存のPostgreSQLインスタンスを使用する場合は、データベースにアクセスするためのホスト名、ユーザー名、ユーザーパスワードを設定する必要があります。

注意、警告メッセージ、およびヒント

ライセンスを確認する

Puppet Enterpriseの購入時に受け取ったlicense.keyファイルで、デプロイ可能なノードの台数を確認します。 Puppet Enterpriseの実行時にログインライセンスの警告メッセージが表示されないようにするには、このファイルを/etc/puppetlabs/license.keyとしてPuppet masterノードにコピーする必要があります。 ライセンスキーファイルがない場合は、sales@puppetlabs.com宛てにメールで再送付を依頼してください。

Puppet Enterpriseは最大10台のノードまで無料でご利用になれます。 ノード台数が10台までの場合、ライセンスキーは不要です。

Puppet Enterpriseのバイナリおよびシンボリックリンク

Puppet Enterpriseのバイナリは/opt/puppetlabs/binにインストールされます。 すべてのユーザーが主要なPuppetツールを使えるよう、インストーラによって、facter、puppet、pe-man、r10k、hiera、およびmcoバイナリのシンボリックリンクが/usr/local/binに自動的に作成されます。 シンボリックリンクは、/usr/local/binが書き込み可能な場合にのみ作成されます。

AIXおよびSolaris 10/11をお使いの場合は、デフォルトパスに/usr/local/binを追加する必要があります。

Mac OS X agentを実行している場合、agentのカタログを適用してPuppetが正常に実行されるまで、シンボルリンクは作成されません。

Puppet EnterpriseがインストールされているPostgreSQLサーバー、PuppetDB、Rubyパッケージと通信するコンポーネントなど、上記以外のPuppet Enterpriseコンポーネントが提供するバイナリのシンボリックリンクは作成されません。 これらのバイナリをデフォルトの$PATHに追加するには、プロファイルに手動で追加するか、またはPATH=/opt/puppetlabs/puppet/bin:/opt/puppetlabs/server/bin:$PATH;export PATHを実行します。
